# CLAUDE.md — Jordan Auto Assistance (JAA) Project

## Project Overview

**JAA (Jordan Auto Assistance)** is a web-based platform that connects stranded motorists
with nearby verified repair shops across Jordan.

When a car breaks down or stops working, the customer does NOT need to know what is wrong.
They simply send a help request with their GPS location and a brief description of the situation
(e.g. "car won't start", "strange noise", "engine light on"). A mechanic from a nearby shop
comes to them, diagnoses the problem on the spot, and either fixes it there or tows it to the shop.

**Tech Stack:**
- ASP.NET Core MVC (.NET 8)
- Entity Framework Core
- SQL Server (database name: `JAADB`)
- ASP.NET Identity (authentication)
- Bootstrap 5 (frontend)
- Razor Views (.cshtml)

---

## Roles

| Role       | Description                                                        |
|------------|--------------------------------------------------------------------|
| Admin      | Manages the entire platform                                        |
| ShopOwner  | Registers a repair shop and responds to customer help requests     |
| Customer   | Sends a help request when their car breaks down                    |

---

## Core Concept

The customer's journey is simple:
1. Car breaks down
2. Customer opens JAA, clicks "Request Help"
3. GPS location is captured automatically
4. Customer writes a short description of the situation (not the technical problem)
5. Request is broadcast to all verified nearby shops
6. A mechanic arrives, diagnoses the car, fixes it on-site OR tows it to the shop
7. After completion, customer leaves a rating for the shop

The customer never selects a service type or diagnoses the problem.
The mechanic decides what needs to be done after arriving.

## Database: JAADB

### Tables & Columns

**ApplicationUsers** (extends ASP.NET Identity IdentityUser)
- Id (nvarchar, PK)
- FullName (nvarchar)
- Email (nvarchar)
- PhoneNumber (nvarchar)
- City (nvarchar)

**RepairShops**
- ShopId (int, PK)
- OwnerId (FK → ApplicationUsers)
- ShopName (nvarchar)
- Description (nvarchar)
- Address (nvarchar)
- City (nvarchar)
- Latitude (float)
- Longitude (float)
- PhoneNumber (nvarchar)
- LogoUrl (nvarchar, nullable)
- IsVerified (bool)
- CreatedAt (datetime)

**ServiceRequests**
- RequestId (int, PK)
- CustomerId (FK → ApplicationUsers)
- ShopId (FK → RepairShops, nullable — null until a shop accepts)
- CustomerLatitude (float)
- CustomerLongitude (float)
- SituationDescription (nvarchar) — what the customer describes e.g. "car won't start"
- DiagnosisNotes (nvarchar, nullable) — filled by the mechanic after arriving
- Resolution (nvarchar, nullable) — what was done: fixed on-site or towed to shop
- Status (int → enum below)
- CreatedAt (datetime)
- UpdatedAt (datetime)

**RequestStatus enum:**
- Pending = 0        (waiting for a shop to accept)
- Accepted = 1       (a shop accepted, mechanic on the way)
- MechanicArrived = 2  (mechanic is at the customer location, diagnosing)
- InProgress = 3     (being fixed on-site or being towed)
- Completed = 4      (car is fixed and returned to customer)
- Cancelled = 5      (cancelled by customer before acceptance)
- Rejected = 6       (no shop accepted within time limit — future feature)

**Payments**
- PaymentId (int, PK)
- RequestId (FK → ServiceRequests)
- Amount (decimal) — set by the shop after diagnosis
- Notes (nvarchar, nullable) — e.g. "replaced battery + labor"
- PaidAt (datetime)

**Feedback**
- FeedbackId (int, PK)
- RequestId (FK → ServiceRequests)
- CustomerId (FK → ApplicationUsers)
- ShopId (FK → RepairShops)
- Rating (int, 1–5)
- Comment (nvarchar, nullable)
- CreatedAt (datetime)

**SystemSettings**
- SettingId (int, PK)
- Key (nvarchar)
- Value (nvarchar)

## Controllers & Views

### AccountController (Public)
- GET/POST Register — role: Customer or ShopOwner
- GET/POST Login — redirect by role after login
- POST Logout

### HomeController (Public)
- GET Index — landing page
- GET About
- GET Contact
- GET Shops — browse all verified shops (no service filter, just city and rating)
- GET ShopDetail(int id) — shop profile, reviews, contact info

### CustomerController [Authorize(Roles="Customer")]
- GET Dashboard
- GET/POST RequestHelp — main action: capture GPS + describe situation
- GET TrackRequest(int id) — see status of active request
- GET History — all past requests
- POST CancelRequest(int id) — only if Status = Pending
- GET/POST Feedback(int requestId)
- GET/POST Profile

### ShopController [Authorize(Roles="ShopOwner")]
- GET Dashboard
- GET/POST ManageShop — edit shop profile
- GET IncomingRequests — all Pending requests in the area
- POST AcceptRequest(int id)
- GET ActiveJobs — accepted requests currently being handled
- POST UpdateStatus(int requestId, RequestStatus status)
- POST AddDiagnosis(int requestId, string diagnosisNotes, string resolution, decimal amount)
- GET CompletedJobs
- GET Earnings
- GET Reviews

### AdminController [Authorize(Roles="Admin")]
- GET Dashboard
- GET/POST Users
- GET/POST Shops — verify/unverify shops
- POST VerifyShop(int id)
- GET Requests — all requests with filters
- GET Reports

## Key Workflows

### Normal breakdown workflow
```
Customer's car breaks down
    ↓
Customer opens RequestHelp page
    ↓
GPS location captured automatically (Leaflet.js + browser geolocation)
    ↓
Customer types a short situation description:
  "Car stopped suddenly", "Won't start", "Overheating", "Flat tire", "Strange noise"
    ↓
ServiceRequest created → Status: Pending, ShopId = null
    ↓
All verified shops see the request in IncomingRequests
    ↓
First shop to accept → ShopId is set → Status: Accepted
    ↓
Mechanic travels to customer GPS location
    ↓
Mechanic arrives → Shop updates → Status: MechanicArrived
    ↓
Mechanic diagnoses the car
    ↓
Shop fills DiagnosisNotes + Resolution + Amount → Status: InProgress
  (Resolution is either "Fixed on-site" or "Towed to shop")
    ↓
Work is done → Shop updates → Status: Completed
    ↓
Customer sees completion and is prompted to leave Feedback (1–5 stars + comment)
```

### Customer cancels before acceptance
```
Customer sees request is still Pending
    ↓
Clicks Cancel → Status: Cancelled (only allowed while Pending)
```

## Important Business Rules

- Customer NEVER selects a service type — there are no service types in this system
- Customer NEVER sees a price before the mechanic arrives and diagnoses
- The price (Payment amount) is set by the shop AFTER diagnosis, not before
- A shop can only accept ONE request at a time per mechanic (simplified: one active job at a time)
- ShopOwner can only manage their own shop — always validate OwnerId == currentUserId
- Feedback can only be submitted once per request and only when Status = Completed
- CancelRequest only works if Status = Pending

## Security

- ASP.NET Identity for auth
- `[ValidateAntiForgeryToken]` on all POST actions
- Role-based `[Authorize]` on all controllers
- ShopOwner can only access their own shop data
- EF Core parameterized queries only

## Coding Conventions

- ViewModels for all views — never pass raw EF models to views
- `async/await` on all DB calls
- All DB access in `/Services/` folder
- TempData["Success"] and TempData["Error"] for messages
- PascalCase for classes/methods, camelCase for variables
